# 第一阶段实现计划

## 1. 背景

本计划基于当前 `AGENTS.md`、`ARCHITECTURE.md`、`docs/企业级RAG知识库入库前治理平台方案设计文档.md`、`docs/architecture/模块边界.md`、`docs/architecture/接口契约.md`、`docs/architecture/数据库设计.md` 和产品文档整理。

第一阶段目标是完成企业级 RAG 知识库入库前治理闭环，不做简单 Demo，也不提前堆复杂能力。

### 2026-06-25 数据库迁移修复记录

- `system-service` 启用 Flyway，确保服务启动时自动执行数据库迁移。
- `V1__init_schema.sql` 对齐已执行数据库记录的 checksum，保留 `knowledge_base` 建表。
- 新增 `V2__create_knowledge_tables.sql` 创建知识库目录表。
- 补充 Flyway MySQL 支持依赖，匹配 Spring Boot 3.3 使用的 Flyway 版本。

### 2026-06-27 文档解析块缺表修复记录

- 新增 `V11__ensure_document_parse_block_table.sql`，幂等补齐 `knowledge_document_parse_block` 表。
- 修复已有环境因历史迁移版本已执行、后续表结构补充不会自动重放导致的文档结构查询缺表问题。

第一阶段主链路：

```text
知识库管理
-> 目录管理
-> 文档上传
-> 文档解析
-> 结构识别
-> 分片生成
-> 向量化
-> ES / Milvus 入库
-> 分片预览
-> 召回测试
-> 审核发布
```

第一阶段默认不启用：

- 多租户。
- 复杂 RBAC 权限。
- 复杂工作流审批。
- BM25 + 向量混合召回。
- Rerank 重排。
- 完整 Agent 工具调用。

## 2. 实现边界

### 2.1 Java 模块

| 模块 | 第一阶段职责 |
|---|---|
| `gateway-service` | 路由、跨域、traceId、鉴权前置预留，不访问业务数据库 |
| `system-service` | 第一阶段核心服务，负责知识库、目录、文档、任务、分片配置、回调、召回测试入口、审核发布 |
| `chat-service` | 保留服务骨架，第一阶段不作为核心交付 |
| `platform-foundation` | 平台基础模块，放统一返回、分页、错误码、业务异常、traceId 上下文等稳定基础能力 |

说明：`platform-foundation` 不放业务逻辑，不放 Controller、Mapper、Service，不访问业务数据库。

### 2.2 Python 模块

| 模块 | 第一阶段职责 |
|---|---|
| `document-process-service` | 消费文档处理任务，执行解析、结构识别、分片，回调 Java |
| `visual-parser-service` | 扫描 PDF、图片 PDF、OCR、版面识别能力入口，第一阶段可先提供可调用骨架 |
| `ai-model-service` | Embedding 调用封装，模型 Provider 预留 |
| `ai-agent-service` | 召回测试执行，调用向量检索和 ES 补全文本 |

### 2.3 `system-service` 包结构

`system-service` 不拆 `system-api`、`system-application`、`system-domain`、`system-infrastructure`、`system-starter` 这类过细模块。第一阶段使用单服务内部分包。

```text
com.aistudio.system
├── controller
├── model
│   ├── request
│   └── response
├── service
│   └── impl
├── mapper
├── entity
├── query
├── convert
├── enums
├── config
├── mq
├── client
├── cache
└── storage
```

### 2.4 编码约束

- Java 包路径统一使用 `com.aistudio`，不增加 `harness` 或其他项目代号层级。
- 知识治理能力归入 `system-service`，不创建 `knowledge-service`。
- Mapper 接口命名使用 `IxxxMapper.java`。
- Mapper XML 命名使用 `XxxMapper.xml`，不加 `I`。
- Mapper XML 路径按业务放置，例如 `resources/mapper/knowledge/KnowledgeBaseMapper.xml`。
- Request、Response 只属于接口层。
- Mapper 不接收 Request，不返回 Response。
- Request 查询数据库前必须转换为 Query。
- Mapper 返回 DO。
- DO 转换为 Response 后返回前端。
- 转换逻辑放在 `convert` 包，优先使用 `builder()`。
- 分页查询使用 MyBatis-Plus `Page<T>` 和 `IPage<T>`。
- ServiceImpl 使用 `@Resource` 注入依赖，不使用构造方法注入。
- Service 入口、异步任务、回调、状态变更、审核发布、异常分支必须记录日志。
- 类、接口、核心方法、复杂逻辑、状态流转、异常分支必须写必要注释。
- Controller 不写业务逻辑。
- DO / Entity 不直接返回给前端。
- 禁止用 `Map` 代替明确对象。

## 6. 任务流规划

### 6.1 文档上传任务流

```text
前端上传文档
-> gateway-service 转发
-> system-service 校验知识库、目录、文件类型、文件大小
-> 计算 file_hash
-> 检查同知识库同目录重复文件
-> 原始文件上传 MinIO
-> 写入 file_resource
-> 写入 knowledge_document
-> 写入 knowledge_document_version
-> 生成 knowledge_process_task
-> 发送 RabbitMQ 消息 DOCUMENT_PARSE_CHUNK
-> 返回 documentId、versionId、taskId
```

### 6.2 文档解析与分片任务流

```text
document-process-service 消费 DOCUMENT_PARSE_CHUNK
-> 下载 MinIO 原始文件
-> Parser Router 判断解析策略
-> 必要时调用 visual-parser-service
-> 生成 document_structure.json
-> 生成 sections.jsonl
-> 根据 chunk_config_snapshot 生成 chunks.jsonl
-> 归档解析产物到 MinIO
-> 回调 system-service
-> system-service 保存解析块和分片元数据
-> 更新文档 parse_status
-> 创建或推进 DOCUMENT_EMBEDDING_INDEX
```

### 6.3 向量化与索引任务流

```text
document-process-service 执行 DOCUMENT_EMBEDDING_INDEX
-> 读取 chunks.jsonl
-> 调用 ai-model-service 生成 embedding
-> 写 Elasticsearch
-> 写 Milvus
-> 归档 embeddings.jsonl 和 process_report.json
-> 回调 system-service
-> system-service 更新 chunk 索引字段
-> 更新文档 index_status
-> 更新任务状态为 SUCCESS
```

### 6.4 任务进度任务流

```text
Python 阶段性回调进度
-> system-service 更新 Redis 当前进度
-> system-service 在关键节点更新 MySQL
-> system-service 通过 SSE 推送进度
-> 前端查询接口兜底
```

### 6.5 召回测试任务流

```text
前端提交测试问题
-> system-service 保存 retrieval_test 记录
-> system-service 调用 ai-agent-service
-> ai-agent-service 调用 ai-model-service 生成 query embedding
-> Milvus TopK 向量召回
-> Elasticsearch 批量补全文本
-> 返回命中分片、来源、页码、相似度
-> system-service 保存 result_json
-> 前端展示召回结果
```

### 6.6 审核发布任务流

```text
文档解析和索引完成
-> 用户提交审核
-> 写入 knowledge_review_record
-> 审核通过
-> 文档进入待发布状态
-> 用户点击发布
-> system-service 修改 MySQL 状态为 PUBLISHING
-> 发送索引状态同步任务
-> Python 更新 ES / Milvus enabled 和 publish_status
-> 回调 system-service
-> system-service 更新为 PUBLISHED
```

### 6.7 失败重试任务流

```text
任务失败
-> 更新 task_status = FAILED
-> 写入 error_code、error_message
-> 记录异常日志
-> 前端展示失败原因
-> 用户点击重新处理
-> system-service 校验当前状态
-> 重新创建或复用任务
-> 重新发送 MQ 消息
```

自动重试最多 3 次，文件损坏、加密 PDF、解析结果异常等问题进入人工处理。

## 8. 交付顺序

### 8.1 第一批：工程骨架和基础设施

交付内容：

- Maven 多模块骨架。
- `platform-foundation` 基础模块。
- `gateway-service` 骨架。
- `system-service` 骨架。
- `chat-service` 骨架。
- Python 四个服务骨架。
- Docker Compose 中间件配置。
- Flyway 初始化入口。
- Knife4j / OpenAPI。
- traceId 日志链路。

验收标准：

- Java 服务可启动。
- Python 服务可启动。
- 网关可转发到 `system-service`。
- 基础健康检查可访问。
- 本地测试脚本可执行。

### 8.2 第二批：数据库和基础 CRUD

交付内容：

- 第一阶段核心表结构。
- DO、Mapper、XML、Query、Convert。
- 知识库管理接口。
- 目录管理接口。
- 分片配置接口。

验收标准：

- 知识库可新增、编辑、分页查询、详情查询、启停。
- 目录可新增、编辑、删除、树形查询。
- 分页查询链路符合 Request -> Query -> DO -> Response。

### 8.3 第三批：文档上传和任务创建

交付内容：

- 文件上传接口。
- `file_hash` 计算和去重。
- MinIO 原始文件存储。
- 文档主表和版本表写入。
- 文档处理任务创建。
- RabbitMQ 消息发送。

验收标准：

- 上传文档后能生成 document、version、task。
- MinIO 能看到原始文件。
- RabbitMQ 能收到 `DOCUMENT_PARSE_CHUNK` 消息。
- 重复文件有明确提示。

### 8.4 第四批：Python 解析回调闭环

交付内容：

- `document-process-service` 消费 MQ。
- Parser Router 基础实现。
- 解析产物归档 MinIO。
- Java 回调接口。
- 解析块和分片元数据入库。
- 任务状态更新。

验收标准：

- 上传一个测试文档后，能生成解析块和 chunk。
- 文档解析状态能从处理中变成成功或失败。
- 失败时有明确错误码、错误信息和日志。

### 8.5 第五批：Embedding、ES、Milvus 入库

交付内容：

- `ai-model-service` Embedding 接口。
- ES chunk 正文索引。
- Milvus 向量写入。
- 索引结果回调。
- chunk 索引字段更新。

验收标准：

- chunk 能写入 ES。
- chunk 向量能写入 Milvus。
- 文档索引状态能正确更新。

### 8.6 第六批：分片预览和进度展示

交付内容：

- 分片分页查询。
- 分片详情查询。
- 分片启停。
- 任务进度查询。
- SSE 任务进度推送。

验收标准：

- 前端或接口可以查看文档分片。
- 文档处理进度可以查询。
- 处理过程中能看到阶段变化。

### 8.7 第七批：召回测试

交付内容：

- 召回测试接口。
- `ai-agent-service` 召回测试能力。
- Milvus TopK。
- ES 批量补全文本。
- 测试记录保存。

验收标准：

- 输入问题后可以返回 TopK 分片。
- 返回结果包含来源文档、目录、页码、相似度、发布状态。
- 召回测试记录可查询。

### 8.8 第八批：审核发布

交付内容：

- 审核驳回。

验收标准：

### 8.9 第九批：测试和验收

## 9. 第一阶段最终验收链路

第一阶段完成后，必须可以跑通：

```text
创建知识库

-> 配置分片规则
-> 上传文档

```

## 10. 风险和待确认问题

| 问题 | 影响 | 处理建议 |
|---|---|---|
| 本地机器是否能跑完整 ES、Milvus、MinIO、RabbitMQ | 影响本地联调 | Docker Compose 先按最小资源配置，必要时分批启动 |
| Embedding 模型是否本地部署 | 影响向量化速度和稳定性 | 第一阶段先支持配置化 Provider，可以先 mock 或本地模型 |
| MinerU 依赖是否稳定 | 影响扫描 PDF 解析 | 先把 Parser Router 和服务边界做好，解析器可替换 |
| ES IK 插件安装方式 | 影响中文检索 | 在 Docker Compose 中固定镜像或安装脚本 |
| 大文件上传限制 | 影响文档入库 | 第一阶段先限制文件大小，后续再做分片上传 |
| 回调安全 | 影响内部接口安全 | 第一阶段校验任务和状态，后续增加签名 |
| MySQL 是否存 chunk 正文 | 影响库体积 | 第一阶段只存预览和对象存储路径，正文放 ES / MinIO |
