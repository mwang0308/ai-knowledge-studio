# 文档解析块逻辑修正计划

## 背景

本次修正聚焦文档处理链路中的解析块逻辑。目标是让 Python 解析流程更贴近以下顺序：

```text
按文件类型生成解析块
-> 每个解析块抽取文字、表格、图片/OCR信息和标题候选
-> 合并所有解析块标题候选
-> 生成整篇目录树
-> 根据目录树生成文档解析块
-> 切细块用于 Embedding
```

## 当前实现差距

- PDF 已经按配置页数拆为 `PdfParseBlock`，默认 2 页一块，并在块内抽取标题、段落、表格、图片和页码元素。
- PDF 目录树已经由文本标题元素合并生成，图片和页码不会参与目录识别，符合契约方向。
- PDF 当前缺少块级 OCR/MinerU 调用；扫描 PDF 或图片型 PDF 只能依赖可提取文本，无法真正补 OCR 文本。
- Word、Markdown、TXT 当前主要直接生成 `section -> block -> chunk`，没有把“解析块/解析块”作为所有格式统一的中间态输出。
- Word 当前按标题样式识别目录，但解析块没有明确记录为“按目录拆出的解析块”。
- 解析产物 `document_structure.json` 已包含 `directory_tree`、`parse_blocks`、`sections`，Java 回调能落库解析块和分片来源。

## 本次修改范围

- 保留既有 Java 与 Python 契约，不新增服务，不改变 MQ 和回调结构。
- Python `document-process-service` 中补齐非 PDF 格式的解析块 ID、解析块名称和来源范围。
- PDF 解析块输出增加标题候选、元素统计、OCR 状态等元数据，供 Java 解析块 `metadata_json` 保存。
- 增加可选 MinerU OCR 调用占位：当环境配置开启并能找到可用适配器时执行；否则记录日志并继续文本解析，不阻塞 MVP 链路。
- 分片继续从正文和表格文本生成，不把未经 OCR 的图片占位写入正文分片。

## 暂不实现

- 暂不把 `visual-parser-service` 做成完整 MinerU 远程解析服务。
- 暂不引入强依赖 MinerU，避免本地环境缺少模型或二进制依赖导致文档处理服务不可启动。
- 暂不重写 Java 表结构；利用现有 `metadata_json` 承载解析块额外元数据。

## 验收点

- PDF 大于配置页数时仍按页码范围生成多个解析块。
- PDF 解析块 metadata 能看到标题候选、元素统计、OCR 状态。
- Word/Markdown/TXT 至少生成可展示的解析块，分片携带 `parse_block_id` 和 `parse_block_name`。
- 图片元素不进入正文分片，未启用 OCR 时不会污染 chunk。
- Python 编译通过，并执行内存级解析校验。
