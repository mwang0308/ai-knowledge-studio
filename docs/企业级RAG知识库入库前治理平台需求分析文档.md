# 企业级 RAG 知识库入库前治理平台需求分析文档

> 文档定位：需求分析 + 竞品调研 + 可行性判断  
> 项目阶段：真实开发前的产品分析文档  
> 目标用户：RAG 项目的开发团队 / AI 平台团队  
> 产品范围：先做“知识入库前治理”，暂不做完整问答端和 Agent 应用编排  
> 对标产品：RAGFlow、Dify、FastGPT 等开源 RAG / AI 应用平台

---

## 1. 项目结论先行

本项目**值得做**，但不建议一开始做成完整的 RAG 平台，更适合先聚焦为一个“企业级 RAG 知识入库前治理平台”。

原因是：

1. 现有开源 RAG 平台普遍已经覆盖知识库、文档上传、分片、检索、问答、工作流等能力，但对“企业内部文档正式入库前的治理流程”支持不够细。
2. 企业 RAG 项目的核心痛点往往不是“能不能问答”，而是“上传进去的知识是否结构正确、分片合理、来源可追溯、发布可控制、召回效果可验证”。
3. RAGFlow 更偏完整 RAG 引擎和文档理解能力，Dify 更偏 AI 应用编排和知识库接入，FastGPT 更偏知识库问答和工作流搭建；三者都不是专门围绕“入库前治理闭环”设计。
4. 如果本项目聚焦“解析结果可视化、结构化分片配置、召回测试、审核发布、版本追溯”，可以形成比较明确的差异化。
5. 对真实开发来说，该项目可以先做成独立模块，也可以后续作为 RAGFlow / Dify / FastGPT 的前置治理层或企业内部 AI 平台的知识治理子系统。

**推荐定位：**

```text
面向企业 AI 平台团队的 RAG 知识入库前治理系统，
用于解决企业文档从上传、解析、结构识别、分片、召回验证、审核发布到可检索上线前的质量控制问题。
```

---

## 2. 项目背景

企业级 RAG 系统不是简单把文件上传到向量库后就能稳定问答。

在真实企业场景中，文档通常存在以下问题：

- Word / PDF / Excel / CSV / Markdown / TXT 等格式混杂。
- PDF 可能是文本型，也可能是扫描件。
- Word 中存在标题、表格、图片、页眉页脚、目录、编号、附件等复杂结构。
- Excel 中存在多 Sheet、合并单元格、多行表头、说明行、空行、跨表关系等问题。
- 企业文档经常有版本，例如制度、流程、费用标准、审批规则等经常更新。
- 文档上传后直接参与检索，容易导致错误知识被问答系统命中。
- 分片质量不可见，开发人员很难判断问答差是模型问题、检索问题、分片问题，还是文档本身质量问题。

因此，在企业 RAG 项目中，需要在正式检索和问答之前增加一层“入库前治理流程”。

本项目围绕以下主线建设：

```text
知识库 → 目录 → 文档 → 文档内容结构 → 知识分片 → 召回测试 → 审核发布
```

该主线来自当前已有原型和人工需求设计，已有需求中已经明确：文档不应直接无组织堆在知识库下，必须挂到具体目录；系统需要支持文档上传校验、结构识别、大文件解析块处理、分片预览、召回测试和审核发布等流程。

## 3. 项目目标

### 3.1 业务目标

本项目的目标不是替代所有 RAG 平台，而是补齐企业 RAG 项目中“入库前治理”的关键能力：

1. 让知识进入正式问答前先被组织、检查、处理、验证和发布。
2. 让开发团队能看到文档解析后的内容结构，而不是只看到上传成功。
3. 让分片结果可预览、可定位来源、可重新处理。
4. 让检索效果在发布前可以通过召回测试验证。
5. 让重要文档必须经过审核后才能正式参与问答。
6. 让文档版本、处理记录、审核记录、发布记录可追溯。

### 3.2 产品目标

产品上建议形成以下能力闭环：

```text
上传前：目录归属、分类标签、分片规则、审核规则
上传中：格式校验、大小校验、重复校验、任务状态
处理后：结构查看、分片预览、异常日志、重新处理
发布前：召回测试、人工审核、发布确认
发布后：下架、版本追溯、重新处理、重新发布
```

### 3.3 技术目标

技术上建议形成一个可独立部署、可集成、可扩展的治理模块：

- Java 后端负责后台管理、权限、任务状态、审核发布、元数据管理。
- Python 服务负责文档解析、结构识别、分片生成、向量化前处理。
- 存储层保留原文、结构化结果、分片结果、任务日志、版本记录。
- 后续可对接 Milvus / Elasticsearch / OpenSearch / RAGFlow / Dify / FastGPT 等系统。

## 4. 目标用户分析

### 4.1 主要用户

本项目的主要用户不是普通问答用户，而是：

```text
RAG 项目的开发团队 / AI 平台团队
```

他们通常负责：

- 企业内部知识库建设。
- 文档解析与清洗。
- 分片策略调优。
- 向量库和检索链路建设。
- 问答效果排查。
- 多业务部门知识接入。
- 知识发布流程管控。

### 4.2 典型用户角色

| 角色 | 关注点 | 需要的能力 |
|---|---|---|
| AI 平台负责人 | 知识是否可控、系统是否稳定、后续能否扩展 | 知识库总览、发布控制、版本追溯 |
| RAG 后端开发 | 分片是否合理、召回是否准确、接口是否好接 | 分片配置、召回测试、处理日志 |
| 文档解析开发 | 文档结构是否识别正确、异常在哪里 | 内容结构查看、解析块记录、异常重试 |
| 业务知识管理员 | 文档是否放对目录、是否审核通过 | 目录管理、审核发布、文档状态 |
| 测试人员 | 发布前效果是否达标 | 测试问题、召回结果、命中来源 |

### 4.3 用户痛点

| 痛点 | 具体表现 | 本项目解决方式 |
|---|---|---|
| 文档上传无治理 | 文件堆在知识库里，目录混乱 | 知识库 + 目录 + 分类 + 标签 |
| 解析结果不可见 | 不知道系统到底解析出了什么 | 文档内容结构可视化 |
| 分片不可控 | 问答差时不知道是不是分片问题 | 分片配置 + 分片预览 |
| 大文件处理不稳定 | 大 PDF / Excel 容易超时或失败 | 解析块处理 + 分段任务 |
| 发布风险高 | 错误知识直接进入正式问答 | 审核发布 + 未发布不可命中 |
| 调试困难 | 无法验证某个问题会命中哪些分片 | 召回测试 |
| 难以追溯 | 不知道某个回答来源于哪个版本文档 | 来源章节、页码、解析块、版本记录 |

## 7. 竞品调研：RAGFlow

### 7.1 产品定位

RAGFlow 是一个开源 RAG 引擎，官方定位强调基于深度文档理解的知识抽取，并支持复杂格式非结构化数据处理。其 GitHub README 中强调 “Quality in, quality out”，包括复杂格式文档理解、模板化分片、可视化文本分片、可追溯引用、多路召回与融合重排等能力。

### 7.2 相关能力

RAGFlow 对本项目最有参考价值的能力包括：

- 深度文档理解。
- 支持复杂非结构化文档。
- 模板化分片。
- 分片可视化，允许人工干预。
- 引用来源可追溯。
- 多路召回和融合重排。
- 支持 Word、PPT、Excel、TXT、图片、扫描件、结构化数据、网页等异构数据源。

### 7.3 优势

| 优势 | 说明 |
|---|---|
| 文档理解能力强 | 更适合复杂文档场景 |
| RAG 链路完整 | 覆盖解析、分片、检索、重排、引用等环节 |
| 工程参考价值高 | 可作为真实开发的重要参考项目 |
| 分片可视化意识强 | 与本项目“分片预览”方向一致 |

### 7.4 可能不足

从本项目角度看，RAGFlow 虽然能力强，但仍有可差异化空间：

1. 更偏完整 RAG 引擎，不是专门做企业内部“入库前治理流程”。
2. 企业目录管理、审核发布、版本治理、目录级规则继承等业务流程可以进一步细化。
3. 如果直接二开 RAGFlow，学习成本和系统复杂度较高。
4. 如果本项目目标是 Java + Python 的企业 AI 平台子模块，完全照搬 RAGFlow 架构不一定合适。

### 7.5 对本项目启发

本项目应学习 RAGFlow 的方向：

- 文档解析结果必须可视化。
- 分片不能是黑盒。
- 召回必须能看到来源和引用。
- 支持复杂文档格式是企业级 RAG 的关键。

但本项目应避免一开始做得过重，建议优先把“治理闭环”做清楚。

## 8. 竞品调研：Dify

### 8.1 产品定位

Dify 是 AI 应用开发平台，覆盖应用编排、RAG 知识库、工作流、模型接入、监控等能力。Dify 文档中明确支持创建知识库、上传文件、配置检索设置、启用引用和归因，并在应用调试预览中输入问题进行调试。

### 8.2 相关能力

Dify 对本项目有参考价值的能力包括：

- Knowledge 知识库模块。
- 上传文件创建知识库。
- 元数据过滤。
- 检索设置。
- 引用和归因。
- Debug and Preview 调试预览。
- Retrieval Testing 检索测试。
- General、Parent-child、Q&A 等分片结构模式。
- 高质量索引、经济索引、向量检索、全文检索、混合检索。

### 8.3 优势

| 优势 | 说明 |
|---|---|
| 应用编排强 | 可以快速把知识库接入聊天应用 |
| 知识库流程完整 | 创建、上传、分片、索引、检索测试都有基础能力 |
| 使用门槛低 | 适合业务和开发人员快速搭建 AI 应用 |
| 检索测试清晰 | 可模拟用户问题测试知识库召回效果 |
| 元数据能力实用 | 支持通过元数据过滤文档 |

### 8.4 可能不足

从本项目角度看，Dify 的主要不足是：

1. 更偏“应用平台”，不是专门做“文档治理平台”。
2. 知识库能力服务于应用构建，入库前审核、目录级规则、版本追溯、解析块处理等能力不是核心重点。
3. 分片模式虽然可选，但对企业复杂文档结构的治理颗粒度仍可进一步增强。
4. 对真实企业的“发布前审批”和“未发布不可命中”流程，仍需要二次开发或外部系统配合。

### 8.5 对本项目启发

本项目应学习 Dify 的方向：

- 召回测试入口要简单直接。
- 分片模式要对用户解释清楚。
- 元数据过滤非常重要。
- 后续可以提供 API，把治理后的知识接入问答应用。

但本项目不应一开始做成 Dify 那种完整 AI 应用平台，否则开发范围会失控。

## 9. 竞品调研：FastGPT

### 9.1 产品定位

FastGPT 是一个基于大模型的知识库问答和 AI Agent 搭建平台，官方介绍中强调数据导入与预处理、知识匹配、智能问答、可视化工作流、API 集成等能力。FastGPT 文档中也强调其知识库系统可以进行数据导入、预处理、知识匹配和智能问答。

### 9.2 相关能力

FastGPT 对本项目有参考价值的能力包括：

- 知识库问答。
- 数据导入和预处理。
- 自动知识结构化。
- 智能数据解析。
- 复杂 PDF 结构处理。
- 图片、表格、公式保留。
- 扫描文件识别。
- Markdown 结构化。
- 工作流编排。
- API 集成。

### 9.3 优势

| 优势 | 说明 |
|---|---|
| 知识库问答落地完整 | 从导入到问答路径较短 |
| 工作流能力实用 | 适合构建业务流程型 AI 应用 |
| 中文生态较强 | 对国内开发者学习和部署友好 |
| 数据处理能力覆盖广 | 支持复杂 PDF、图片、表格、公式等处理方向 |

### 9.4 可能不足

从本项目角度看，FastGPT 的主要不足是：

1. 更偏“快速搭建知识库问答应用”，不是专门做“文档入库治理”。
2. 对目录级审核、发布可见性、文档版本治理、解析块处理过程可视化等企业治理能力还可以进一步细化。
3. 对开发团队排查“解析错在哪里、分片为什么这样切、召回为什么命中这个片段”的专门调试能力，可以做得更强。

### 9.5 对本项目启发

本项目应学习 FastGPT 的方向：

- 数据导入体验要简单。
- 解析与预处理要尽量自动化。
- 后续可以保留工作流和 API 集成扩展空间。

但本项目的核心不应是“快速问答搭建”，而应是“高质量知识进入问答系统之前的治理”。

## 11. 市场与机会判断

### 11.1 为什么还有机会做

虽然 RAGFlow、Dify、FastGPT 已经比较成熟，但它们解决的是更大的问题：

- RAGFlow：如何构建强大的 RAG 引擎。
- Dify：如何快速搭建 AI 应用。
- FastGPT：如何快速落地知识库问答和工作流。

本项目解决的是更窄但更真实的企业问题：

```text
企业文档进入正式 RAG 检索系统之前，如何保证它是可组织、可解析、可分片、可测试、可审核、可追溯的？
```

这个问题在真实项目中很常见，尤其是：

- 财务制度知识库。
- 人事制度知识库。
- 法务合同知识库。
- 客服知识库。
- 运维手册知识库。
- 产品文档知识库。
- 企业内部流程规范知识库。

这些场景不允许错误知识直接进入问答系统。

### 11.2 差异化机会

本项目可以形成以下差异化：

| 差异化方向 | 说明 |
|---|---|
| 入库前治理闭环 | 从上传到发布前完整管控 |
| 目录级规则继承 | 目录可配置默认分片规则和审核规则 |
| 分片可解释 | 每个分片能看到来源章节、页码、解析块 |
| 解析块中间态 | 大文件处理过程可观察、可重试 |
| 召回测试前置 | 发布前先验证召回效果 |
| 审核发布隔离 | 未发布内容不进入正式问答 |
| 版本追溯 | 文档、分片、审核、发布均有记录 |
| 面向开发团队 | 强调调试、排查、集成，而不是只给业务人员使用 |

### 11.3 是否值得做

结论：**值得做，但必须控制边界。**

不建议做：

- 完整 AI 应用平台。
- 完整 Agent 编排平台。
- 完整聊天问答系统。
- 从零实现所有文档解析算法。
- 一开始就做复杂权限、租户、流程引擎。

建议做：

- 入库前治理主流程。
- 文档结构查看。
- 分片配置和预览。
- 召回测试。
- 审核发布。
- 处理日志和异常重试。
- 基础版本追溯。

## 12. 产品定位

### 12.1 一句话定位

```text
一个面向企业 AI 平台团队的 RAG 知识入库前治理平台，用于在知识正式参与问答前完成文档组织、解析检查、结构化分片、召回验证、审核发布和版本追溯。
```

### 12.2 不做什么

为了保证真实开发可落地，当前阶段明确不做：

1. 不做完整聊天问答前台。
2. 不做复杂 Agent 工作流编排。
3. 不做业务低代码平台。
4. 不做复杂 BPM 审批流，只做轻量审核。
5. 不做通用文档协作编辑。
6. 不做从零训练 Embedding / Rerank 模型。
7. 不做自研完整 OCR / 版面识别模型，优先集成成熟工具。

### 12.3 要做什么

当前阶段重点做：

1. 知识库管理。
2. 目录管理。
3. 文档上传和校验。
4. 文档解析任务管理。
5. 文档内容结构展示。
6. 大文件解析块处理记录。
7. 分片配置。
8. 分片预览。
9. 召回测试。
10. 轻量审核发布。
11. 重新处理。
12. 基础版本和操作记录。

```text

  ↓

  ↓

  ↓

## 14. 核心功能需求

## 14.1 知识库管理

### 功能说明

知识库是企业知识空间的最高管理单元。

### 核心字段

| 字段 | 说明 |
|---|---|
| 知识库名称 | 如财务共享知识库、人事制度知识库 |
| 知识库说明 | 说明用途 |
| 默认分片配置 | 新目录和文档默认继承 |
| 默认审核规则 | 是否默认需要审核 |
| 状态 | 启用 / 停用 |
| 文档数量 | 当前知识库文档总数 |
| 已发布数量 | 正式可检索文档数 |
| 待审核数量 | 需要处理的审核任务 |

### MVP 建议

第一阶段必须做：

- 新增知识库。
- 编辑知识库。
- 知识库列表。
- 进入知识库详情。
- 启用 / 停用。

第一阶段可暂缓：

- 多租户隔离。
- 复杂权限矩阵。
- 知识库容量计费。

## 14.2 目录管理

### 功能说明

目录表示文档放在哪里，不等同于分类和标签。

### 核心能力

- 新增目录。
- 编辑目录。
- 删除空目录。
- 拖拽排序。
- 移动目录。
- 设置负责人。
- 设置目录默认分片配置。
- 设置目录默认审核规则。
- 点击目录筛选文档。

### 关键规则

1. 文档必须挂到具体目录。
2. 有文档的目录不允许直接删除。
3. 目录规则默认影响后续新上传文档。
4. 已有文档是否应用新规则，需要手动重新处理。

## 14.3 文档上传与校验

### 支持格式

| 类型 | 格式 | MVP 建议 |
|---|---|---|
| Word | doc、docx | 必须支持 docx，doc 可后置 |
| PDF | pdf | 必须支持文本 PDF，扫描 PDF 可后置 |
| Markdown | md、markdown | 必须支持 |
| TXT | txt | 必须支持 |
| Excel | xls、xlsx | 必须支持 xlsx，xls 可后置 |
| CSV | csv | 必须支持 |
| PPT | ppt、pptx | 后续扩展 |

### 上传校验

- 文件格式是否合法。
- 文件大小是否超限。
- 文件是否为空。
- 文件名是否合法。
- 是否重复上传。
- 是否选择知识库。
- 是否选择目录。
- 是否满足目录规则。

### 上传字段

| 字段 | 是否必填 | 说明 |
|---|---|---|
| 所属知识库 | 是 | 当前知识空间 |
| 所属目录 | 是 | 文档归属位置 |
| 文档分类 | 否 | 制度、流程、FAQ、标准表 |
| 标签 | 否 | 辅助检索 |
| 分片配置 | 否 | 默认继承目录配置 |
| 是否审核 | 否 | 默认继承目录审核规则 |
| 审核人 / 审核组 | 条件必填 | 需要审核时填写 |
| 文件 | 是 | 支持单个或批量 |

## 14.4 文档处理任务

### 任务阶段

```

### 任务记录

| 字段 | 说明 |
|---|---|
| 任务 ID | 唯一任务标识 |
| 文档 ID | 关联文档 |
| 任务类型 | 解析、分片、向量化、重新处理 |
| 当前阶段 | 当前处理进度 |
| 进度百分比 | 用于页面展示 |
| 开始时间 | 任务开始时间 |
| 结束时间 | 任务结束时间 |
| 失败原因 | 失败时记录 |
| 日志摘要 | 处理日志 |

## 14.5 文档内容结构

### 功能说明

文档内容结构是解析后的“可读结构”，用于让开发团队理解文档被系统识别成了什么。

### 不同文档展示方式

| 文档类型 | 展示方式 |
|---|---|
| Word 有标题 | 章节树 + 段落 + 表格 |
| Word 无标题 | 段落组 + 表格 + 图片 |
| PDF 有目录 | 章节树 + 页码 |
| PDF 无目录 | 页码范围 + 内容块 |
| 扫描 PDF | 页码 + OCR 文本 + 图片区域 |
| Markdown | 标题树 + 表格 + 代码块 + 引用 |
| Excel | Sheet + 表头 + 数据行范围 |
| CSV | 表头 + 行范围 |
| TXT | 段落组 / 弱标题结构 |

### 价值

该功能是本项目区别于普通文件上传系统的重要功能。它能帮助开发团队判断：

- 标题识别是否正确。
- 表格是否被保留。
- 页码来源是否准确。
- Excel 表头是否识别正确。
- Markdown 层级是否保留。
- 扫描 PDF OCR 是否可靠。

## 14.6 大文件解析块处理

### 功能说明

解析块是大文件处理过程中的中间单元，不等于最终知识分片。

### 解析块拆分依据

| 文件类型 | 解析块依据 |
|---|---|
| PDF | 页码范围 |
| Word | 标题层级、段落范围、页码范围 |
| Markdown | 标题层级 |
| TXT | 段落范围、文本大小 |
| Excel | Sheet、行范围 |
| CSV | 行范围 |

### 解析块列表字段

| 字段 | 说明 |
|---|---|
| 解析块编号 | block_001 |
| 拆分方式 | 页码范围、Sheet、标题范围 |
| 来源范围 | 第 1 - 20 页、Sheet1 第 1 - 500 行 |
| 状态 | 待处理、处理中、成功、失败 |
| 生成分片数 | 当前解析块生成多少分片 |
| 操作 | 查看详情、关联分片、重试 |

### 设计原则

1. 普通用户不需要理解解析块。
2. 解析块主要服务开发和排障。
3. 分片预览不应强依赖解析块入口。
4. 解析块失败时应支持局部重试。

  ↓

## 14.9 召回测试

### 功能说明

召回测试用于验证知识在正式发布前是否能被正确检索命中。

### 测试范围

| 范围 | 说明 |
|---|---|

| 当前目录 | 测试目录下文档 |
| 当前知识库 | 测试整个知识库 |
| 未发布内容 | 发布前验证 |
| 已发布内容 | 模拟正式问答 |

### 结果字段

| 字段 | 说明 |
|---|---|
| 排名 | Top 1、Top 2 |
| 命中分片 | chunk_id |
| 命中内容 | 内容摘要 |
| 来源文档 | 文件名 |
| 来源目录 | 目录名称 |
| 来源章节 | 章节 / Sheet / 页码 |
| 相似度 | 检索分数 |
| 操作 | 查看分片、定位来源 |

### MVP 建议

第一阶段召回测试不需要做复杂评测体系，先支持：

- 输入单个问题。
- 返回 TopK 分片。
- 显示来源和分数。
- 支持按文档、目录、知识库范围测试。

## 14.10 审核发布

| 状态 | 是否管理端可见 | 是否正式问答可命中 |
|---|---|---|
| 处理完成 | 是 | 否 |
| 待审核 | 是 | 否 |
| 审核通过 | 是 | 否 |
| 已发布 | 是 | 是 |
| 审核驳回 | 是 | 否 |
| 已下架 | 是 | 否 |
| 处理失败 | 是 | 否 |

### 审核动作

- 审核通过。
- 审核驳回。
- 填写审核意见。

待审核 → 审核通过 / 审核驳回

| 功能 | 暂缓原因 |
|---|---|
| 多租户 | 初期复杂度较高 |

| 流程引擎审批 | 轻量审核足够 |
| 批量召回评测 | 第二阶段更合适 |
| 分片 A/B 实验 | 需要数据积累 |
| 多模态图片检索 | 成本和复杂度较高 |
| PPT 深度解析 | 优先级低于 Word/PDF/Excel |
| 在线编辑分片 | 容易引入数据一致性问题 |
| 自动生成问答对 | 可作为增强功能 |

### 15.3 MVP 验收标准

MVP 完成后，应能支撑如下场景：

```text
管理员创建一个财务共享知识库，
在下面创建“报销制度 / 费用标准 / 审批流程”等目录，
上传 Word、PDF、Excel 文档，
系统完成解析和分片，
管理员可以查看文档结构和分片结果，
输入一个测试问题验证召回命中，
审核通过后发布，
发布后该文档才可进入正式问答检索范围。
```

## 16. 亮点功能设计

## 16.1 亮点一：入库前治理闭环

普通 RAG 项目通常只强调：

```text
上传 → 分片 → 向量化 → 问答
```

本项目强调：

```text
上传 → 校验 → 结构识别 → 分片预览 → 召回测试 → 审核发布 → 正式可检索
```

这更符合企业真实场景。

---

## 16.2 亮点二：目录级规则继承

目录不仅是文件夹，还可以承载规则：

- 默认分片配置。
- 默认审核规则。
- 默认负责人。
- 默认分类建议。

这样可以让不同业务目录使用不同入库策略，例如：

| 目录 | 分片策略 | 审核规则 |
|---|---|---|
| 报销制度 | 按标题 + 段落 | 必须审核 |
| 费用标准 | 表格分片 | 必须审核 |
| 常见问题 | Q&A 分片 | 可自动发布 |
| 操作手册 | 按章节分片 | 需要审核 |

---

## 16.3 亮点三：解析块作为大文件处理中间态

大文件处理经常失败，如果没有解析块，开发团队只能看到“处理失败”。

引入解析块后，可以看到：

- 哪个页码范围失败。
- 哪个 Sheet 失败。
- 哪个章节失败。
- 哪个解析块生成了多少分片。
- 是否可以局部重试。

这对真实开发排障非常有价值。

---

## 16.4 亮点四：分片来源可解释

每个分片必须能追溯到：

- 来源知识库。
- 来源目录。
- 来源文档。
- 来源版本。
- 来源章节。
- 来源页码。
- 来源解析块。

这样可以解决企业 RAG 常见问题：

```text
AI 这个回答到底来自哪份文件？
是不是旧版本？
是不是未发布内容？
是不是命中了错误目录？
```

---

## 16.5 亮点五：发布前召回测试

召回测试不是问答聊天，而是检索调试工具。

它可以帮助开发团队判断：

- 这个问题能否命中正确分片。
- Top1 是否正确。
- 相似度是否偏低。
- 命中内容是否来自正确目录。
- 是否需要调整分片策略。
- 是否需要补充标签或元数据。

---

## 16.6 亮点六：未发布内容隔离

企业知识库最大的风险之一是错误知识、旧知识、未审核知识进入正式问答。

因此需要明确：

```text
管理端可见 ≠ 正式问答可命中
```

只有已发布内容才能进入正式检索范围。

---

## 16.7 亮点七：面向开发团队的处理日志

普通业务系统只需要展示“成功 / 失败”。

本项目面向 RAG 开发团队，应展示更多排障信息：

- 文件解析耗时。
- 结构识别耗时。
- 分片数量。
- 向量化耗时。
- 失败阶段。
- 异常摘要。
- 重试次数。
- 解析工具版本。
- 分片配置版本。

## 18. 关键风险分析

### 18.1 文档解析质量风险

风险：

- PDF 表格识别不稳定。
- 扫描件 OCR 错误率高。
- Word 复杂格式丢失。
- Excel 合并单元格语义难恢复。

应对：

- MVP 先支持常见格式和常规结构。
- 对复杂文件展示“解析置信度 / 异常提示”。
- 保留原文下载和人工检查入口。
- 支持重新处理和替换解析工具。

---

### 18.2 分片策略复杂度风险

风险：

- 分片策略过多，用户不知道怎么选。
- 配置项太细，开发成本高。
- 不同格式规则差异大。

应对：

- 第一阶段只提供模板化配置。
- 默认给出推荐策略。
- 支持试切分预览。
- 高级配置后置。

---

### 18.3 召回效果评估风险

风险：

- 单次召回测试不能代表整体效果。
- 相似度分数不同模型不可直接比较。
- 用户可能误以为召回测试等于最终问答质量。

应对：

- 页面明确说明“召回测试只验证检索命中，不代表最终生成回答”。
- 第二阶段增加批量评测问题集。
- 支持人工标注命中是否正确。

---

### 18.4 范围失控风险

风险：

- 做着做着变成 Dify。
- 做着做着变成完整 RAGFlow。
- 做着做着又要 Agent、工作流、聊天前台。

应对：

- 当前只做入库前治理。
- 问答端只做召回测试。
- Agent 和应用发布作为后续扩展。

---

### 18.5 性能风险

风险：

- 大文件解析慢。
- 批量上传占用资源。
- 向量化调用慢。
- 多人同时处理任务阻塞。

应对：

- 上传和处理异步化。
- 大文件解析块分段处理。
- 任务队列限流。
- 分阶段保存处理结果。
- 支持失败重试。

## 23. 最终判断

### 23.1 想法可靠性

该想法可靠。

原因：

1. 企业 RAG 项目确实存在入库前治理痛点。
2. 当前已有开源平台并未完全覆盖企业内部审核发布和治理流程。
3. 当前原型已经具备清晰主线，不是空想功能。
4. 技术上可通过 Java + Python + 文档解析工具 + 检索组件实现。
5. MVP 范围可以控制，不必一开始做完整 RAG 平台。

### 23.2 项目是否值得做

值得做，但建议按以下原则推进：

```text
先做窄，做深；
不要做大而全；
先解决入库前治理；
后续再扩展问答、Agent、工作流。
```

### 23.3 推荐落地路径

第一步：完成 MVP。

```text
知识库 / 目录 / 文档上传 / 解析状态 / 分片预览 / 召回测试 / 审核发布
```

第二步：增强文档处理能力。

```text
解析块详情 / 重新处理 / 版本记录 / Excel 和 PDF 优化
```

第三步：增强质量评估能力。

```text
批量召回测试 / 标准问题集 / 分片质量评分
```

第四步：开放集成。

```text
对接 Milvus / ES / Dify / RAGFlow / FastGPT / 企业问答前台
```
